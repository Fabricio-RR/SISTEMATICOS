## 3. Lo que se hizo (detalle)

### 3.1. Fix crítico — el dashboard mostraba "No se pudo conectar con el servidor"
- **Causa real:** el endpoint `/api/partidos/` devolvía **HTTP 500** porque el modelo `Torneo`
  usaba la columna `estado_previo` (función de pausar/reactivar torneos) que **no existía en la
  base de datos** — faltaba la migración.
- **Solución:** migración **0016** (`estado_previo` en `torneos`).
- El mensaje del frontend era engañoso: el backend sí estaba activo; una de las 5 llamadas del
  dashboard fallaba y tumbaba todo el bloque.

### 3.2. Landing pública (`app/page.tsx`)
- **Scroll-spy**: el menú (Inicio / Clasificación / Brackets / Resultados) resalta con la línea
  roja la sección activa según el scroll o el clic.
- **Scroll suave** (`globals.css`).
- Nueva sección **"Últimos Resultados"** que consume partidos finalizados del backend.

### 3.3. Deportes — gestión completa
- **Backend:**
  - `PATCH /api/deportes/{id}` para editar y activar/desactivar.
  - Parámetro `?incluir_inactivos=true` para que el admin vea los desactivados.
  - `DELETE` convertido en **borrado lógico** (`estado = 2`): se oculta del listado pero se
    conserva en la BD. Bloquea la eliminación si hay **equipos o torneos asociados**.
  - Migración **0017** (`deportes.estado`: 1 vigente, 2 eliminado).
- **Frontend** (`app/admin/deportes`):
  - Botones **Editar** (lápiz), **Activar/Desactivar** (power) y **Eliminar**.
  - **Modal de confirmación** propio antes de eliminar (no se usa `window.confirm`).

### 3.4. Equipos — gestión
- **Backend:**
  - `DELETE` convertido en **borrado lógico** (`estado = 'eliminado'`): se oculta del listado
    pero se conserva en la BD; mantiene el bloqueo por **atletas o inscripciones**.
  - Migración **0018** (permite `estado = 'eliminado'` en `club_equipo`).
- **Frontend** (`app/admin/equipos`):
  - Botón **Editar** (renombrar) y **Eliminar** con **modal de confirmación**.
  - Método `updateEquipo` en el cliente (usa el `PUT` existente).
  - *Nota:* por decisión del usuario, en Equipos **no** se agregó activar/desactivar.

### 3.5. Limpieza de repositorio
- Se dejó de trackear `frontend/.next-dev/` y `frontend/.next-build/` (artefactos de build) y se
  agregaron al `.gitignore`.

### 3.6. Auditoría de deuda técnica (diagnóstico)
- **Backend:** arquitectura por capas correcta (config / core / models / schemas / routers /
  services), config centralizada, rate-limiting, `joinedload` para evitar N+1.
- **Frontend:** componentes gigantes (equipos = 1236 líneas, 42 `useState`), sin primitivas UI
  compartidas, duplicación (`inputCls` ×6, badges ×7), y **sin capa de datos**.
- **Escalabilidad:** principal hallazgo → **sin paginación** (31 listados con `.all()`).

### 3.7. Capa de datos del frontend — TanStack Query (punto #3 de la auditoría)
- **Base:** `app/providers.tsx` (`QueryClient` global) + `lib/hooks.ts` (hooks de lectura +
  *query key factory* `qk`).
- **Pantallas migradas** (de `useEffect/cargar()/useState` a hooks con caché compartida):
  Dashboard, Deportes, Instituciones, Torneos, Inscripciones, Atletas, Resultados, Equipos y la
  Landing pública.
- El **dashboard** refresca en vivo con `refetchInterval` y cada métrica es independiente: si una
  llamada falla, las demás siguen mostrándose (corrige de raíz el aviso global).
